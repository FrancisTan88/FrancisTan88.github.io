Data Structure - Least Recently Used Cache

基本介紹

LRUC(Least Recently Used Cache)是一種實現快取的資料結構，概念上是將越少用到的資料放在越尾端(tail)，越常用到的則放在越前面的位置(head)，且通常會有最大資料容量的限制(capacity)，一旦超過此容量就會從「最少使用到的資料」開始刪除，如此來提升資料取用的效率，而實作上通常會使用「doubly linked list」和「hashmap」。一個典型的LRUC如下：

先備知識

Doubly Linked List

  1. 觀念：
  • 相較於singly linked list操作insert(), remove()等operation需要O(n)，doubly linked list可以降到O(1)，而在實作上通常會搭配hashmap(keys: node address)一起使用。
  • 然而需要使用比singly linked list更大的空間來儲存hashmap以及「previous links」。
  • 所以使用時機為，當我們較不在乎空間複雜度，且希望降低與search有關(e.g. insertion(), remove()等等)的時間複雜度時。
  1. 實作：
  • 在實作上為了操作方便，一般會初始化並maintain head, tail兩個pointer。
  • 由ListNode組成，每個node主要包含：
    • key: 用來索引到(透過hashmap)該node。
    • value: node實際儲存的值。
    • previous: 與前一個node的link。
    • next: 與後一個node的link。
  • 常見的operation主要包含：
    • remove(key: int): 刪除key值為”key”的node，做法上和singly linked list一樣都是「將removed node的前一個接到removed node的下一個」，差別在於要再對反方向做一樣的事情，處理完list之後記得去hashmap pop掉該key值。
    • insert_after(insert_key: int, key: int, value: int): 將「key值為”key”、值為”value”的node插在key值為”insert_key”的node後面」，操作上為先將inserted node的”後半部”接好，再接inserted node的前半部，因為若先前半部就會失去後半部的pointer(i.e. 無法成功insert)，最後要注意，由於是用hashmap在儲存整個list nodes的位址且我們期望達到O(1)indexing，這邊不容許key值有duplicates，所以在insert完node後若有duplicate，我們需要將其從list刪除，最後再更新hashmap。

實作

其實LRUC實質上就是doubly linked list，只是附加了priority和capacity的特性，常見的operation主要包括：

  • get(key): 返回索引值為”key”的value(若存在)，且執行此操作時同時意味該value為「最近被用到的值」，所以要將其重新插到最前面(i.e. head.next)的位置。

  • put(key, value): 將索引為”key”、值為”value”的node插入list，同樣地，執行此操作時同時意味該value為「最近被用到的值」，所以一定是插在最前面(i.e. head.next)的位置。

    • 若key值本不存在：直接插入即可。
    • 若key值存在(Note: 我們不允許有duplicates)：將原本的node刪除再插入。
    • 更新hashmap。
    • 檢查最大容量：在上述情況中，若是key值本不存在，相當於我們新增了一個node(而非覆蓋某個原有的node)，此時若nodes數量超出最大限制(i.e. len(hashmap) > capacity)，我們要將「最不常用到的值」刪除(i.e. tail.prev)。

Time complexity: O(1), for each operation.
Space complexity: O(n), which is the number of nodes.
